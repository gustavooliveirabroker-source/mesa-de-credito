01 · Reaproveitamento

O que já existe e a Mesa herda

A leitura do repo cashin-cashout-hub confirmou que a fundação está pronta. Verde = reaproveitado; roxo = novo da Mesa.

REUSA Auth & papéis

user_roles + useProfile() + guards de rota. A Mesa adiciona mesa_members com papéis próprios (analista, aprovador N1–N3…).

REUSA Carteira & cobrança

debit_wallet_atomic() — débito atômico idempotente por ref_id. A análise é cobrada sem risco de duplicidade.

REUSA Fornecedores de dados

queries + query_modules (CPF/CNPJ, score, SCR, jurídico…). mesa_consultations apenas referencia a consulta executada.

REUSA White-label

white_label_configs + tenant_wl_id + wl_owner_config_id(). A Mesa nasce com a coluna — o "sistema com a marca do cliente" vem de graça.

REUSA UI & PDF

app-shell, responsive-table, ui/* (shadcn) e o renderer pdf-lib do parecer. Mesma identidade.

NOVO Motor & esteira

Política configurável, Kanban, faixas X/Y/Z, comitê/alçadas, agente financeiro e histórico do subject — as tabelas mesa_*.

06 · Decisões & riscos

Pontos que precisam do seu OK

Lovable no main. A Lovable dá git add -A no main sozinho. Solução: toda a Mesa vive na branch feat/mesa-credito, migration aditiva (não toca tabelas existentes), merge coordenado quando você decidir. Evita conflito com o editor web.
Como aplicar a migration. O Supabase é gerido pela Lovable. Precisamos definir: (a) colar o SQL no SQL Editor do Supabase, ou (b) deixar a Lovable aplicar ao commitar o arquivo em supabase/migrations/. Recomendo (a) num ambiente de teste primeiro.
Cobrança. Cada análise debita a carteira do cliente via debit_wallet_atomic('client', client_id, custo, 'query_debit', 'mesa_analise', analysis_id) — idempotente. Preço dos produtos (Essencial/Empresarial/Completa) entra como itens de tabela de preço.

Próximo passo

Se o schema e o plano estiverem de acordo, eu crio a branch feat/mesa-credito, coloco a migration e começo a Fase 1 (MVP) — proposta → política → comitê fim-a-fim.
